🧠 Single source of truth

One brain for your whole product.

Product Brain keeps specs, vocabulary, domains, and decisions in one place — connected to your code across every repo, answerable by Claude, and authored in whatever method your team already uses.

Knowledge stops being a guest inside each code repo. It becomes its own product: a hub that holds what your team knows, pulls in your code repos, and builds a single knowledge graph you can ask anything.

┌────────────────────────────────────┐ │ your hub repo │ ← single source of truth │ constitution · vocabulary · domains│ │ docs/ (specs, decisions, notes, …) │ │ graph/ (built over docs + code) │ └───────────────┬────────────────────┘ pb sync │ pulls code, builds one graph ┌───────────────┴───────────────┐ ▼ ▼ ┌─────────────┐ ┌──────────────┐ │ web-app │ │ backend-api │ ← your code repos │ (untouched) │ │ (untouched) │ (nothing added) └─────────────┘ └──────────────┘

Two things, kept separate

🛠️

Product Brain — the framework

The open-source tooling: the brainify skill, hub templates, and these docs. It operates on a hub; it never lives inside one.

📚

Your hub — an instance

Your team's actual knowledge, in its own git repo. Contains constitution, vocabulary, domains, docs, and the graph — and no skills.

This separation is what keeps the framework method-neutral and your hub clean and diffable.

How it works

One command assembles everything and builds a single graph.

The reframe

A. Knowledge is the product

It lives in a dedicated hub repo — the single source of truth. Code repos are just sources the hub reads.

B. The framework ≠ the hub

Tooling ships separately from any team's knowledge. The framework operates on a hub from the outside.

pb sync — one graph over one workspace

Because the hub pulls and builds centrally, there's no per-repo merging to manage. One graphify run spans your docs and all your code:

pb sync
  1. pull / refresh each repo from brain.config.json → .work/repos/<id>
  2. run graphify over a workspace =
       { constitution.md, vocabulary.md, domains.md, docs/**, .work/repos/** }
  3. write graph/graph.json + graph/GRAPH_REPORT.md

In the resulting graph: code structure comes free from tree-sitter; and graphify's semantic edges plus Leiden communities softly co-locate related code and docs across repos — so the booking spec, the Appointment model, and the SessionCard component tend to land together, with no manual linking.

App repos stay untouched. Registration lives only in the hub's brain.config.json. No files are added to your web or API repos — zero adoption friction.

Anatomy of a hub

A required core, an extensible docs/ tree, and the graph. The hub is its own git repo.

acme-brain/ ← the hub: its own git repo brain.config.json repos to pull · doc types · graph settings constitution.md principles / non-negotiables [REQUIRED] vocabulary.md glossary: product ↔ code [REQUIRED] domains.md area → owner, status, repos [recommended] docs/ extensible knowledge (Markdown-first) specs/ decisions/ meeting-notes/ research/ <your-type>/ workflows/ role lenses: pm, backend, frontend, qa… graph/ graph.json + GRAPH_REPORT.md (pb sync) .work/ gitignored: pulled code, sync-time only

Required core required

Two files are what make a directory a "brain" — the knowledge that can't be derived from code:

constitution.md

The non-negotiable principles every change must respect. Optional per-repo notes where a repo legitimately differs.

vocabulary.md

The glossary tying product language to code in each repo — the bridge between "Session" in a meeting and Appointment in the code.

Domains recommended · graph-assisted

domains.md maps functional areas to owners, status, and repos. It's recommended rather than required because it's largely derivable: after a sync, the graph's communities are good candidate domains, so you curate names and owners instead of authoring from scratch.

Extensible docs your choice

Register any doc types you want in brain.config.json and drop knowledge into docs/<type>/. Ship with specs, decisions, meeting-notes, research, runbooks — or invent your own. graphify connects it all automatically through inferred edges and communities — no manual cross-linking needed.

{
  "repos": [
    { "id": "backend-api", "url": "…", "src": ["app/"] },
    { "id": "web-app",     "url": "…", "src": ["src/"] }
  ],
  "doc_types": ["specs", "decisions", "meeting-notes", "research"],
  "graph": { "out": "graph/" }
}

Method-agnostic by design

Product Brain prescribes structure, not methodology.

There is no required way to write a spec. Use plain Markdown, RFCs, Shape Up pitches, Spec Kit, or paste an export from Notion. The only requirement is that a spec is a Markdown file under docs/specs/ that links to the domain and terms it touches. Bring the tool your team already likes.

What the framework asks of you

  • Keep the required core current.
  • Put knowledge under docs/ (Markdown preferred).
  • Run pb sync so the graph stays fresh.

What it never dictates

  • Your spec format or process.
  • Your approval ceremony.
  • Which doc types you must keep.

The shipped spec-template.md is a convenience, not a mandate — delete it and use your own.

Graph & formats

The graph is built by graphify. What you can graph drives the format choices.

Markdown-first, JSON only for config

Prefer Markdown for knowledge you author and maintain — it's diffable and free to ingest. JSON is used only for machine config (brain.config.json), which Claude reads directly and never needs in the graph. There are no manual links to keep up: graphify infers edges and clusters related docs and code into communities on its own. (Native artifacts like PDFs and recordings are also supported — see below.)

Why a graph outside the repo still answers questions

Every graph node stores a source_file pointer. When Claude answers, it traverses the graph and opens only the few chunks that matter — it doesn't re-read the codebase. So the graph can sit in the hub, away from the code, and still return precise chunks; the only requirement is that the pulled source in .work/ is present (it is, until the next sync). graphify reports a ~70× token reduction per query versus reading raw files on a mixed corpus.

It's multi-modal — drop in recordings, PDFs, designs

You don't have to convert everything to Markdown. graphify ingests far more than text, in three passes over one assembled workspace:

PassHandlesHowCost
1 — Code25 languages + SQL schemastree-sitter AST, on your machineFree, local
2 — Voice / videomeeting recordings, calls (.mp3 .wav .mp4 .mov)transcribed locally with faster-whisper, seeded with your top concepts for domain focusFree, local
3 — Docs / imagesMarkdown, PDF, .docx/.xlsx, imagesread by Claude subagents into graph nodesCosts tokens (first build)

So a recorded kickoff call or a PDF brief can live in the hub and become queryable knowledge — the meeting's decisions end up in the same graph as the code they affect. Markdown stays the preferred format for knowledge you actively author and maintain (it's diffable and free of token cost), but native artifacts are first-class inputs.

Performance

The first pb sync pays the extraction cost (code is free; docs/images cost tokens; audio/video is local). Every file is fingerprinted by content hash, so later syncs skip everything unchanged and only re-process what moved. There is no separate merge step — it's a single graph from a single run over the assembled workspace.

Setup guide

From nothing to a queryable brain in a few steps.

1

Install graphify

pip install graphifyy      # CLI is `graphify`; no LLM key needed for code
graphify --version
2

Create the hub repo

Make a new git repo for the hub — it's your single source of truth and benefits from PR review and history.

3

Run brainify

Open the hub in Claude (Cowork or Claude Code) and say "Set up product brain." It audits what's present, then walks you through declaring repos, writing the required core, mapping domains, registering doc types, and building the graph — one step at a time.

4

Sync

pb sync     # pulls your repos, builds one graph over docs + code

Now ask Claude about your product, your code, or your decisions.

Your application repos require no structural changes — registration lives only in the hub's brain.config.json.

Optional: make an app repo hub-aware

If you want a Claude session running inside todo-api or todo-web to benefit from the hub automatically, paste the app-repo snippet (templates/app-repo-claude-md-snippet.md) into that repo's CLAUDE.md. It's the only thing you ever add to an app repo, it's pure guidance (no config, no dependency), and it tells Claude to:

  • check the hub's vocabulary.md so it uses the product's real names;
  • read the relevant spec and domain before answering "what should this do / why";
  • respect the hub's constitution.md;
  • prefer the hub's graph for cross-repo questions, and suggest pb sync if it's stale.

The result: in-repo Claude answers get more accurate and efficient, without duplicating any knowledge into the code repo.

Example: a to-do app

A complete, tiny hub for a two-repo product — todo-api and todo-web. It ships in this framework under examples/todo-app/.

todo-brain/ brain.config.json todo-api + todo-web · doc types constitution.md per-user scoping · validate at edge · soft delete · tests gate merge vocabulary.md Task · List · Due date (product ↔ code) domains.md Tasks · Lists docs/ specs/001-create-task.md decisions/ADR-001-soft-delete.md meeting-notes/2026-06-17-kickoff.md

The glossary bridges product and code

One vocabulary entry ties the product word to what each repo calls it:

## Task
A single thing the user wants to get done.
- Also called: todo, item, card
- In code: `Task` (todo-api model), `TaskItem` (todo-web component)

One spec, both repos

The create-task spec declares the repos it touches and tags each acceptance criterion by layer, so backend, frontend, and QA all read the same source:

ID: 001-create-task   Domain: Tasks   Repos: todo-api, todo-web

Acceptance
  (api)  POST /tasks with a title → 201, scoped to the current user
  (api)  empty title → 422 field error
  (web)  Enter in quick-add adds the task to the top, no reload
  (web)  failed create shows an inline error, keeps the typed text

Decisions and meetings are knowledge too

The "soft delete" decision (ADR-001) and the kickoff meeting note live in the same hub and graph as the code — so "why can't we hard-delete?" has a documented answer, and the reasoning from the kickoff isn't lost. Ask Claude "how does creating a task work across both apps?" and it answers from the spec, the vocabulary, and both codebases at once.

Open problems

Deliberately deferred — tracked, not yet built. deferred

Cross-repo linking adapters

Wiring the web graph to the API graph via the contract between them (OpenAPI / GraphQL / tRPC / path heuristics). Hard because the contract style varies per product and heuristic matching is error-prone.

Ripple / impact analysis

Precise "change X here → these things break there," especially across repos. Depends on the linking above. Soft co-location via graph communities is the interim answer.

Freshness via CI-push

Moving graph builds from central pull into each repo's CI so the hub is always current. An optimization, not required for v1.

Vocabulary ↔ graph linter

Auto-verify that each glossary "in code" reference still resolves to a real node after a sync.

Non-git knowledge sources

Pulling docs from Notion / Drive into the hub as exported Markdown.

FAQ

Do I have to use Spec Kit?

No. Product Brain is method-agnostic. Spec Kit is one option among many; use any spec format that produces Markdown under docs/specs/.

Is the hub a separate repo?

Yes — its own git repo. That gives version history, PR review of knowledge, access control, and a clean directory for graphify to build over.

Does anything get added to my app repos?

No. The hub pulls them and registers them centrally in brain.config.json. An optional one-line CLAUDE pointer is the only thing you might add, and nothing depends on it.

Can the graph answer if it lives outside the code repo?

Yes. Nodes carry source_file pointers, so the agent traverses the graph and opens just the relevant chunks. The graph's location is irrelevant as long as the pulled source is present.

Do I have to link documents together manually?

No. graphify infers semantic edges and clusters related docs and code into communities on its own, so there are no wikilinks to author or maintain. Just write clear Markdown and let the graph connect it.

Can it tell me what breaks across repos if I change something?

Not precisely yet — that's the deferred ripple/adapter work. Today you get grounded querying and soft cross-repo grouping via graph communities.

Can I add my own doc types, like meeting notes?

Yes — that's a first-class feature. Add the type to brain.config.json and drop Markdown into docs/<type>/.
