Section 2

The Problems — In Depth

Each of these problems is chronic, costly, and solvable. Here's how each one manifests and what it actually costs.

📄
Problem 1

Documentation Gap

Specs don't exist, or they were written during an early sprint and never updated. The product has shipped four major features since then, and the document still describes the initial prototype. Nobody trusts it — so nobody reads it — so nobody updates it. A vicious cycle.

This isn't a discipline problem; it's a tooling problem. Writing specs in a vacuum, disconnected from the actual code, means the spec decays the moment the first line is merged. There's no mechanism to surface drift, no prompt to update, no connection between the spec and the thing it describes.

🔍 You know this is happening when...

A new developer asks "where's the spec for patient authentication?" and the answer is "there isn't one" or "there was one but it's wrong." Onboarding takes weeks instead of days. Every new hire has to read the code from scratch.

💸 Cost of not solving it

Onboarding time 3–6x longer than it should be. Feature regressions because requirements were never formally captured. PMs writing specs based on assumptions rather than ground truth. Engineers building the wrong thing because no one aligned on what "done" means.

🔬 Real example

The supervision feature — assigning supervisors to therapists — had no spec. We reverse-engineered it from code and discovered rules that lived only in the original developer's head: a therapist can only have one active supervisor, reassignment creates an audit trail, and the old supervisor relationship isn't deleted but archived. None of this was documented anywhere.

🧠
Problem 2

Institutional Knowledge Loss

The engineer who built the authentication system left. The PM who defined the initial clinical workflows is now at a different company. The design decisions that shaped the data model were made in a two-hour whiteboard session that nobody photographed. That knowledge is gone.

What remains is archaeology. Future engineers reverse-engineer intent from code. They guess at why a particular constraint exists. They remove it because they don't understand it — and something breaks in production six weeks later for a reason no one can immediately explain.

🔍 You know this is happening when...

Engineers regularly say "I don't know why this works this way" or "let me check with [person who has left]." PRs introduce changes that break undocumented invariants. Postmortems frequently conclude with "we didn't know that constraint existed."

💸 Cost of not solving it

Production bugs caused by violating implicit business rules. Incorrect decisions due to missing context. Slow delivery because engineers are afraid to change code they don't understand. Every team member departure is a knowledge loss event.

🔬 Real example

The TreatmentPlanStatus::COMPLETED enum value was defined in the code — it existed as a valid state — but no business logic ever transitioned a plan to that state. The state had been planned, the code stub was written, but the implementation was never completed. It sat there for months. No one knew. The graph query surfaced it in 30 seconds.

🔍
Problem 3

Codebase Understanding

A new developer joins and faces 500 files. Or an existing developer needs to implement a feature in a domain they've never touched. Where do they start? Which files are central to the domain? Which are utilities? Which are legacy cruft that everything depends on but nobody understands?

The standard answer — "read the code" — doesn't scale past about 50 files. At 500 files, reading top-to-bottom is weeks of work. Code search helps but returns isolated snippets without structural context. You find the class but not why it exists or what it connects to.

🔍 You know this is happening when...

Developers take 2–3 days to understand the blast radius of a change. PRs regularly miss touching all the relevant files. New feature implementation accidentally duplicates existing logic because the developer didn't know it existed.

💸 Cost of not solving it

Slow feature delivery. Duplicated code. Bugs caused by incomplete implementations. Senior engineers spending 30% of their time answering "where does X work?" questions that should be self-serve.

🔬 Real example

The graphify god-node analysis of a healthcare scheduling API immediately identified Appointment, User, and TreatmentPlan as central nodes with 30+ connections each. A new developer can see in 30 seconds that these three models are the core of the system — and which services, actions, and policies touch them.

🔤
Problem 4

Vocabulary Mismatch

The clinical team calls it a "session." The database model is Appointment. The mobile app displays "Visit." The API response key is meeting_id. These are all the same concept — but a new developer reading the docs and the code simultaneously has to manually build this translation table in their head.

When vocabulary fragments across apps, documentation, and stakeholders, communication breaks down. A PM writing a spec uses "session." The developer implements it as a new model because they didn't realize Appointment was the same thing. A bug is introduced not from bad code but from a vocabulary gap.

🔍 You know this is happening when...

PRs are regularly accompanied by Slack threads clarifying "what do you mean by X." Spec reviews take twice as long because participants are using different words for the same thing. New models get created that duplicate existing ones under a different name.

💸 Cost of not solving it

Misimplemented features. Duplicated data models. UI copy inconsistency. Hours of clarification meetings that should never need to happen. Cross-team communication overhead that grows with every new stakeholder.

🔬 Real example

In a healthcare platform, clinical staff consistently use "session" when referring to therapy appointments. The codebase uses Appointment. Without a vocabulary map, every new developer and every PM writing a spec has to independently discover this mapping — usually at the cost of at least one misunderstanding along the way.

🌊
Problem 5

Ripple Effect

A product decision changes. The "Supervision" workflow is being redesigned. Which other specs does this touch? Which code areas are affected? Which test cases need to be updated? In a disconnected system, the answer to each question requires a separate investigation.

The dangerous assumption is that a change is local when it isn't. A vocabulary change cascades through every spec that uses that term. A core model change cascades through every service that references it. Without structural mapping, these cascades are invisible until something breaks.

🔍 You know this is happening when...

Features shipped without updating related specs. Integration bugs because two teams changed interconnected behavior independently. Post-release bug reports for features that "weren't changed" but depended on something that was.

💸 Cost of not solving it

Silent regressions. Incomplete migrations. Stale test cases that pass but no longer test the right thing. Cross-team coordination failures that surface only after release.

🔬 Real example

If the supervision assignment model changes — say, allowing multiple supervisors instead of one — the ripple touches: the Supervision spec, the Patient Access spec (which controls what a supervisor can see), the Appointment spec (supervisor visibility of sessions), and every API endpoint that returns supervisor data. The graph surfaces all of these connections immediately.

🔗
Problem 6

Multi-App Queries

Your product is three apps: a Laravel API, a React Native mobile app, and a React web dashboard. A question like "when a therapist cancels an appointment, what happens across all three apps?" spans all three codebases. Without a unified knowledge layer, answering it requires three separate investigations and manual synthesis.

Teams work around this by having app-specific experts — "ask Sarah about the mobile app, ask Ahmed about the API." This creates knowledge silos that slow down cross-functional work and create single points of failure when those people are unavailable.

🔍 You know this is happening when...

Cross-app features require coordination calls just to understand the current state. Developers working on one app don't know what the other apps assume or depend on. Inconsistent behavior across apps that should be synchronized.

💸 Cost of not solving it

Cross-app bugs. Integration failures. Slow delivery of cross-cutting features. Over-dependence on specific engineers who know a specific app. API contract violations discovered in QA instead of design.

🔬 Real example

A merged graph of the API backend + the mobile app reveals which mobile screens use which API endpoints, which shared status enum values must stay in sync, and which shared business terms (like Appointment) appear in both codebases. Cross-app questions become single queries instead of multi-day investigations.

Section 3

The Solution Stack

Four components, each solving a distinct class of problem. Together, they form a complete product intelligence layer.

Core Principle

"Graph navigates, Claude reasons."

1

Graphify

pip install graphifyy

AST-based knowledge graph builder. Parses your codebase with Tree-sitter — no LLM needed for code files. Produces an interactive graph.html and machine-readable graph.json. Identifies god nodes, community clusters, surprising cross-domain connections, and dead code patterns.

2

Speckit

.specify/ convention

Structured spec framework. A directory convention + markdown templates + governance rules. The constitution enforces consistency. Templates enforce completeness. Speckit skills in Claude Cowork automate the spec-writing workflow — grounded in your actual code, not invented from thin air.

3

Vocabulary Map

.specify/vocabulary.json

Machine-readable synonym dictionary. Maps business terms to code terms to UI terms. Single source of truth for the entire team. Processed by graphify as synonym edges — so a query for "session" returns results for Appointment. PM-owned, developer-readable, machine-queryable.

4

Claude

Reasoning Layer

The reasoning layer that ties everything together. Claude uses BFS traversal of the graph to find relevant nodes, reads 2–5 targeted files, applies vocabulary translation, and synthesizes answers grounded in real code. It doesn't hallucinate structure it can't verify — it navigates structure that actually exists.

Section 4

Setup Guide

Five phases. Total time from zero to fully operational: approximately half a day for a single repo, one day for multi-repo.

1

Graph Your Codebase

AST-based parsing — no LLM needed

~30 min

Install graphify and run it against your source directory. For a Laravel project, point it at the app/ directory. The --no-label flag skips semantic labeling (which requires an LLM key) and uses structural analysis only — perfectly sufficient for codebase navigation.

pip install graphifyy
cd your-project/
graphify app/ --out graphify-out --no-label
# Opens graph.html in your browser for interactive exploration
# Also produces: graphify-out/graph.json, GRAPH_REPORT.md
ℹ️
LLM key not required for code files. Tree-sitter parses source code locally — zero data leaves your machine. An LLM API key is only needed if you want graphify to semantically extract content from docs, PDFs, or images in a /docs folder.

After running, open graph.html in your browser. Look for:

  • God nodes — large nodes with many connections. These are your core domain models.
  • Community clusters — groups of closely connected nodes. These are your feature domains.
  • Isolated nodes — small components with few connections. May be dead code or utilities.
  • Surprising bridges — nodes that connect otherwise separate clusters. Often architectural seams.
2

Initialize Speckit Governance

Directory structure + constitution + templates

~1 hour

Create the .specify/ directory structure at the root of your project. This is the governance layer — it defines your project's principles, canonical vocabulary, and spec templates.

.specify/ memory/ constitution.md ← project principles & canonical vocabulary templates/ spec-template.md ← feature spec structure plan-template.md ← implementation plan structure tasks-template.md ← task breakdown structure init-options.json ← {"feature_numbering": "sequential"} feature.json ← tracks currently active feature specs/ ← all feature specs live here

The constitution is the most important file. It must define:

  • Project identity — what this system does and who it serves
  • 4–6 architectural principles — non-negotiable rules that every spec and implementation must follow
  • Canonical vocabulary section — official terms and their aliases (before vocabulary.json exists)
  • Governance rules — how specs are written, reviewed, and amended
Example principle

"Every HTTP endpoint MUST be implemented as a single-purpose Action class. No logic in controllers. No logic in route files."

3

Build the Vocabulary Map

Machine-readable synonym dictionary

~30 min

Create .specify/vocabulary.json. This bridges the gap between business language, UI copy, and code terms. The PM owns it; the developer reads it; graphify processes it as synonym edges.

// .specify/vocabulary.json
{
  "terms": [
    {
      "canonical": "Appointment",
      "code_terms": ["Appointment", "AppointmentModel"],
      "business_terms": ["session", "visit", "meeting"],
      "ui_terms": ["Session", "Visit"],
      "notes": "Always 'session' in clinical context. 'Appointment' in API."
    },
    {
      "canonical": "TreatmentPlan",
      "code_terms": ["TreatmentPlan"],
      "business_terms": ["care plan", "therapy plan", "program"],
      "notes": "Clinical teams say 'care plan'. Code uses TreatmentPlan."
    }
  ]
}
💡
Auto-extract draft: Ask Claude to scan the graph god nodes + any existing documentation and produce a draft vocabulary.json for PM review. Takes 5 minutes to generate a starting point you can then refine.
4

Write Your First Spec

AI-grounded spec generation from real code

~1 hour

Use the speckit-specify skill in Claude Cowork. This workflow produces a spec that is grounded in your actual codebase — not invented from thin air.

# In Claude Cowork, say:
"I want to add supervisor assignment to therapist profiles.
A clinic admin should be able to assign a supervisor to a therapist,
and the supervisor should be able to see all sessions for their assigned
therapists."

# Claude will:
# 1. Query the graph for relevant nodes (User, Therapist, Appointment, etc.)
# 2. Ask 4-5 clarifying questions based on what it finds
# 3. Write a grounded spec referencing actual classes and methods
# 4. Generate a checklist for the implementation

The spec will reference real class names, existing patterns, and actual data models — because Claude queried the graph before writing it.

5

Multi-Repo Setup (if needed)

Merge graphs for cross-app queries

~15 min

If your product spans multiple repositories (API + mobile + web), graph each separately and then merge them. This lets you query across all apps while still being able to rebuild individual graphs independently when one app changes.

graphify merge-graphs \
  api/graphify-out/graph.json \
  mobile/graphify-out/graph.json \
  --out merged/graph.json

# Rebuild strategy: only rebuild the app that changed
# Then re-run the merge with the updated graph
graphify app/ --out api/graphify-out --no-label  # after API changes
graphify merge-graphs api/graphify-out/graph.json mobile/graphify-out/graph.json --out merged/graph.json
ℹ️
Point Claude to merged/graph.json for cross-app questions. Point it to individual graphs for app-specific questions. The vocabulary.json is shared — keep it in a central location accessible to both repos.
Section 5

Role Guides

Each role interacts with the stack differently. Here's what each person owns, does daily, and is responsible for maintaining.

Daily

  • Ask questions about the product in Claude Cowork: "how does X work?", "what does Y do?", "where does Z happen?"
  • Use vocabulary.json terms when writing tickets, specs, or user stories — always use canonical terms
  • When a feature doesn't match expectations, query the graph before assuming it's a bug

Weekly

  • Review speckit-converge output for any active features — check for spec drift from implementation
  • Review open [NEEDS CLARIFICATION] markers in specs and resolve them before the sprint ends
  • Verify that shipped features match their spec's success criteria

Per Feature

  • Run speckit-specify workflow before development begins — never write spec after implementation
  • Use speckit-clarify to resolve ambiguities before the developer picks up the ticket
  • After implementation: run speckit-converge to validate the spec matches what was built
  • Add new vocabulary terms discovered during spec writing to vocabulary.json

Reverse-Engineering an Existing Product

  • Ask Claude to extract feature domains from graph community clusters and god nodes
  • For each domain: Claude reads the code, prepares 5 WHY questions, PM answers in 10 min
  • Claude writes the spec combining code evidence with PM answers — PM reviews and signs off
  • Work domain by domain: Authentication → Appointments → TreatmentPlans → Supervision → etc.

Ownership

  • Own the constitution — you are its author, gatekeeper, and primary amender
  • Version-control the constitution: every amendment is a PR, reviewed before merging
  • Own vocabulary.json alongside the PM — you validate that code terms are correct
  • Own the graph rebuild schedule and tooling integration (CI hooks if applicable)

After Significant Code Changes

  • Trigger a graph rebuild when: model renamed, major feature added, directory restructured, new repo added
  • Review GRAPH_REPORT.md after rebuild — check for unexpected new god nodes or broken community structure
  • Run speckit-analyze before major releases to surface spec/implementation gaps

Quarterly

  • Constitution review: do the principles still reflect how we build? Amend and bump version if needed
  • Vocabulary audit: any new code terms that have no business equivalent? Any deprecated terms still in the map?
  • Full graph rebuild from scratch (purges stale cache, re-parses everything)
  • Archive completed feature specs — keep specs/ clean and current

During PRs

  • Check that PRs introducing new domain models also update vocabulary.json
  • Verify that PRs implementing specced features don't deviate from spec without explicit amendment
  • Flag PRs that add new architectural patterns not covered by the constitution — potential amendment trigger

Before Starting a Feature

  • Read the spec. If there's no spec, ask for one before writing a line of code
  • Query the graph to understand existing code in the feature's domain
  • Ask Claude: "what classes and services are involved in [domain]?" to get a map before diving in
  • Check vocabulary.json for canonical terms in the domain — use them in your code and comments

During Implementation

  • When you discover something that contradicts the spec, flag it immediately — don't silently deviate
  • When adding a new domain concept, add it to vocabulary.json alongside the implementation
  • Use the constitution's architectural principles as a checklist — every new endpoint needs an Action class

After Implementation

  • Run speckit-converge to validate your implementation against the spec
  • Review the converge output — gaps are decisions, not failures. Resolve each one explicitly
  • Trigger a graph rebuild if you renamed models, added major new files, or restructured directories

UI Copy and Terminology

  • Use vocabulary.json as the single source of truth for UI copy — always use the canonical business term in UIs
  • Before finalizing UI copy, verify the term exists in vocabulary.json (or add it with PM approval)
  • When clinical/domain experts use a different term than what's in the UI, escalate to vocabulary.json — don't decide alone

Flow Changes

  • When changing a user flow, review the spec for that feature before finalizing designs
  • Use speckit-converge to understand what the current implementation does — designs should reflect reality or explicitly override it
  • Flag designs that require behavior changes — those need spec amendments, not just design changes

New Features

  • Query the graph (via Claude) for what data and concepts exist before designing new flows
  • Review the vocabulary map before designing new UI labels — don't invent terms that conflict with existing canonical ones
  • Contribute to spec writing for features you're designing — the UX insight belongs in the spec's "context" section

Test Scenario Generation

  • Ask Claude: "generate test scenarios for [feature name] based on its spec" — point to the spec and graph
  • Claude will use actual class names and method signatures from the graph — not invented ones
  • Use the spec's success criteria section as your acceptance test checklist, item by item

After Spec Changes

  • Query the merged graph for areas affected by the changed spec — don't test only the obvious paths
  • Review vocabulary.json if a spec change involved renaming or redefining a canonical term
  • Re-run speckit-converge after spec changes to identify implementation gaps that may be missed by tests

Edge Case Discovery

  • Ask Claude: "what are the edge cases for [feature]?" — Claude can reason from graph structure + spec constraints
  • Use god-node analysis to find high-risk areas: models with many connections fail in unexpected ways when changed
  • Use the graph to identify integration points between features — these are the highest-risk test areas
Section 6

Workflows — How-Tos

Step-by-step recipes for the six most common tasks this system enables.

1
"How does feature X work?"
  1. Ask Claude in natural language
    No need for technical framing. "How does appointment cancellation work?" or "When is a treatment plan marked complete?" work perfectly.
  2. Claude queries graphify
    Claude uses BFS traversal from the relevant concept node — following edges to find the classes, services, events, and policies involved.
  3. Claude reads 2–5 targeted files
    Rather than reading 500 files, Claude reads only the files identified by the graph traversal as directly relevant. Precise, not exhaustive.
  4. Synthesized answer with file references
    The answer includes the actual class names, method names, and file paths involved — grounded in reality, not guesswork.
Real example: "When is TreatmentPlan status changed to COMPLETED?" → Graph traversal revealed no transition logic to COMPLETED state existed. The enum value was defined but never triggered. Found in 3 queries, 30 seconds.
2
"Write a spec for a new feature"
  1. Describe the problem context to Claude
    Tell Claude what you want to accomplish and why. Include any known constraints. Don't worry about technical detail — that's what the graph is for.
  2. Claude queries the graph for relevant existing code
    Before asking any questions, Claude maps the relevant domain in the codebase — understanding what models, services, and patterns already exist.
  3. Claude asks 4–5 clarifying questions
    Questions are grounded in what the graph revealed. Examples: "The graph shows Therapist and User are separate models — should supervisor assignment go on the User or Therapist model?", "Cardinality: can one therapist have multiple supervisors simultaneously, or only one?"
  4. speckit-specify generates grounded spec + checklist
    The spec references real class names, uses canonical vocabulary terms, follows constitution principles, and includes a task checklist grounded in actual files that need changing.
  5. speckit-clarify for remaining gaps
    Any open questions or [NEEDS CLARIFICATION] markers are resolved in a second pass before development begins.
3
"Reverse-engineer specs for an existing product"
  1. Extract feature domains from graph structure
    Claude analyzes the graph's community clusters and god nodes to produce a domain list: Authentication, Appointments, TreatmentPlans, Supervision, Billing, etc.
  2. For each domain: code read → WHY questions
    Claude reads the code for a single domain, then prepares 5 WHY questions that only the PM or business owner can answer. "Why can a therapist only have one active supervisor?" "Why does cancellation preserve the record instead of deleting it?"
  3. PM answers (10 min per domain)
    A focused conversation, not a marathon. Claude has already done the code analysis — the PM just supplies business rationale.
  4. Claude writes spec combining code evidence + PM answers
    The resulting spec captures both WHAT the code does and WHY — the institutional knowledge that would otherwise live only in the original developer's memory.
  5. Architecture Decision Record (ADR) created alongside spec
    Key design decisions are captured in a companion ADR — preserving the reasoning so future developers understand why the code is the way it is.
4
"A spec changed — what's affected?"
  1. Describe the change to Claude
    "We're changing supervision to allow multiple active supervisors per therapist instead of one." That's enough context to start the analysis.
  2. Claude queries merged spec graph for shared concept nodes
    Traversal from the changed concept finds all other specs that reference the same concept — surfacing the full ripple before any code changes.
  3. Impact list: specs, code areas, test cases
    Three-category output: which specs need updating, which code areas need changing, which test cases are now invalid or need extension.
  4. Review constitution for vocabulary changes
    If the change involves a term rename or redefinition, update vocabulary.json and check if the constitution needs amendment.
  5. Update affected artifacts
    Update specs, then code, then tests — in that order. Use speckit-converge to verify alignment after each update.
5
"Generate test scenarios for a feature"
  1. Point Claude to the spec's success criteria
    The success criteria section of a well-written spec is a direct source of acceptance test cases. Claude reads it first.
  2. Claude cross-references with code graph
    For each scenario, Claude finds the actual class names, method signatures, and data models involved — so tests reference real code, not imagined APIs.
  3. Test scenarios generated with real signatures
    Output includes: scenario name, preconditions, action, expected outcome, and the specific class/method being tested — grounded in the actual implementation.
Why this matters: AI-generated test scenarios without graph context invent class names that don't exist. With graph context, every scenario references AppointmentCancellationAction, not a hallucinated CancelAppointmentService.
6
"Multi-app question"
  1. Ensure merged graph exists
    Run graphify merge-graphs to produce merged/graph.json. Point Claude's context at the merged graph.
  2. Ask the question normally
    "When a therapist cancels a session in the mobile app, what happens in the API and what does the web dashboard show?" Claude routes to the right sub-graph for each part.
  3. Cross-app connections surface via shared concept names
    The merged graph links the mobile SessionCard component to the API's AppointmentController via shared concept nodes — connections that would be invisible looking at either repo alone.
Section 7

Hygiene & Maintenance

A system that isn't maintained decays. Here's the minimum viable maintenance schedule to keep the stack accurate and useful.

Per Code Change

☐Rebuild affected app's graph (incremental via cache — seconds, not minutes)
☐Run speckit-converge if a spec exists for the changed area
☐Add new domain terms introduced in this change to vocabulary.json

Per Feature

☐Run speckit-specify before implementation begins — spec first, code second
☐Run speckit-clarify to resolve open questions before development picks up the ticket
☐Run speckit-converge after implementation to validate spec/code alignment
☐Update vocabulary.json with any new canonical terms introduced by the feature

Per Sprint

☐Review all open [NEEDS CLARIFICATION] markers in active specs — resolve or defer explicitly
☐Check for spec drift: run speckit-converge across all active specs, not just the most recent one
☐Update CHANGELOG with major product decisions made during the sprint

Per Quarter

☐Constitution review: do the principles still reflect how we build? Amend and version-bump if needed
☐Vocabulary audit: find new code terms with no business equivalent; find deprecated terms still in the map
☐Rebuild all graphs from scratch — purges stale incremental cache, ensures full accuracy
☐Archive completed feature specs — move to specs/archive/ to keep active specs/ clean

Graph Rebuild Triggers (Mandatory)

These events invalidate the existing graph and require a rebuild before the graph is queried again.

  • Model class renamed or moved to a different directory
  • New major feature area added (new directory under app/)
  • Significant directory restructure or namespace change
  • New app or repository added to the multi-repo family
  • Major refactor that changes inter-class dependency structure
⚡
Incremental rebuilds are fast. The graphify cache makes re-runs on unchanged files nearly instant. Only changed files are re-parsed. A full rebuild of a 500-file project takes ~60 seconds. An incremental rebuild after a typical PR takes under 10 seconds.

Section 9

Why This Approach

Every tool choice in this stack was preceded by at least one alternative that didn't work well enough. Here's the honest reasoning.

Our Starting Point

We started with a running API serving a clinical team. The product had been in production for months. No specs existed. No vocabulary alignment between the clinical language used by the team and the code terms used by the developers. No documentation of why design decisions were made. The only source of truth was the code itself, and the code had no narrative.

The question was: how do you close that gap without rewriting everything, without a six-month documentation project, and without disrupting the current development pace?

Problem 1 — Codebase Understanding

AlternativeWhy it fell short
Vector embeddings / RAG (Cursor, GitHub Copilot) Good for autocomplete and finding similar code. Poor for "explain the whole system" — returns chunks without connected structure. Can't answer "what are all the services that touch Appointment?"
Manual code reading Doesn't scale past ~50 files. At 500 files, reading top-to-bottom takes weeks. No persistent artifact — every new team member starts from scratch.
README-driven docs Outdated within weeks. Requires ongoing human discipline to maintain. Still doesn't show structural connections between components.

Why graph: BFS traversal gives precise, connected answers. The graph knows which classes depend on which. It doesn't just find a class — it shows you the neighborhood. We found the COMPLETED state gap in 3 queries, 30 seconds. Manual reading would have taken hours, if the gap was ever found at all.

Problem 2 — Documentation

AlternativeWhy it fell short
Confluence/Notion templates Require sustained discipline to fill out and keep current. Not connected to code — no mechanism to surface drift. Work well for process docs, not feature specs.
Auto-generated API docs (Swagger) Covers endpoint signatures, not product intent. "POST /appointments" doesn't tell you why certain appointment types require supervisor approval. Covers WHAT, not WHY.
AI doc generation (dump code → ask GPT) Produces shallow summaries. Without business context from the PM, it documents implementation details without product rationale. Misses institutional knowledge.

Why speckit: Structured templates enforce completeness. The constitution enforces consistency. The PM interview workflow captures WHY — not just WHAT the code does. The result is a spec that a developer, a PM, and a new engineer can all read and trust.

Problem 3 — Vocabulary Mismatch

AlternativeWhy it fell short
DDD Ubiquitous Language Ideal in greenfield. In running production code, renaming models is high-risk database migration territory. Not practical without a freeze and a migration sprint.
Flat glossary markdown Human-readable but not machine-queryable. Decays as fast as any other doc. Can't be processed by graphify as synonym edges.
Inline spec definitions Fragmented. Every spec defines terms locally. No single source of truth. Definitions drift across specs over time.

Why vocabulary.json: Machine-readable. Processed by graphify as synonym edges — so a query for "session" resolves to Appointment. Single source of truth owned by the PM. Low maintenance overhead. Survives model renaming without code changes.

Problem 4 — Ripple Effect

AlternativeWhy it fell short
Change management tickets Manual process. Misses implicit structural dependencies. Depends on the developer knowing what to check. Works for known dependencies, fails for unknown ones.
Full test coverage Catches behavioral regressions but not spec-level changes. A test can pass while a spec constraint is violated. Complementary, not sufficient.
Architecture diagrams Static. Outdated within a sprint if not manually maintained. Show intended architecture, not actual code structure.

Why merged graph + speckit-converge: Structural ripple is detected automatically via shared concept nodes in the graph. Behavioral gaps are caught by tests. Together, they cover both structural and behavioral impact — each layer doing what it does best.

Problem 5 — Multi-App Queries

AlternativeWhy it fell short
Shared Confluence space Still siloed by app in practice. No code connections — just human-written descriptions that may or may not reference each other correctly.
Super-folder graphify (clone all repos into one dir) Produces the same result as merge but loses per-app query isolation. Rebuilding one app requires re-running on all of them.

Why separate + merge: Graph each app independently so you can rebuild only what changed. Query individual graphs for app-specific questions. Query the merged graph for cross-cutting questions. The merge step is cheap; the rebuild isolation is valuable.

What We Didn't Solve (Honest)

  • Behavioral correctness — whether logic inside a function is correct. The graph shows structure, not semantics. Unit tests are still necessary and complementary.
  • Semantic vocabulary drift — same word, different meaning across two specs. The constitution and human review process catch this, but the system doesn't automate it.
  • Automated test generation — the system generates test scenarios, but a human still needs to validate that those scenarios are correct and cover the right edge cases.

Timeline of Conclusions

How the framework emerged from a real session on a production product.

  • Built the first graph
    Ran graphify on the API codebase. Immediately identified Appointment, User, and TreatmentPlan as god nodes with 30+ connections each. Community clustering revealed the feature domains without any prior knowledge.
  • First query — OTP authentication
    "How does OTP authentication work?" — Claude traversed the graph from the User node to find the OTP model, the verification action, and the token expiry logic. Answer in 30 seconds, grounded in real file paths.
  • Found the COMPLETED state gap
    "When is TreatmentPlanStatus changed to COMPLETED?" — Graph traversal found the enum definition, found all the places status was set, and found no transition to COMPLETED. A stub state with no business logic. Had been in the codebase for months unnoticed.
  • Built speckit
    Recognized that the graph answers WHAT but not WHY. Designed the speckit convention to capture business rationale alongside code structure. Constitution, templates, and the PM interview workflow emerged from this need.
  • Wrote the supervision spec
    Used the speckit-specify workflow to write the first full grounded spec for supervisor assignment. Graph-grounded clarifying questions surfaced the cardinality ambiguity (one supervisor vs. many) before a line of code was written.
  • Vocabulary discussion
    Realized "session" and "Appointment" were causing miscommunication between the clinical team and developers. Designed vocabulary.json as a machine-readable bridge. Extended graphify to process synonym edges.
  • Ripple framework
    Asked "if we change supervision rules, what else is affected?" — discovered three other specs touched the same concept. Formalized the merged-graph + speckit-converge pattern for ripple analysis.

Open problem cards

Each card is an invitation. If you have a solution — a template, a skill step, an integration, a workflow — open a PR. See the Contribute section for how.

📥

Unstructured feature intake

Stakeholder ideas arrive as Slack messages, emails, and Notion pages with no consistent format. There's no way to know if an idea is already specced, in-flight, or was rejected last quarter. Product Brain starts at spec writing — but ideas need a home before that.

Proposed approach

A lightweight intake template (idea title, problem statement, who asked, business case) that Claude can parse to check for existing specs and draft a pre-spec summary.

⚖️

No prioritization layer

Specs get written, but there's no framework connecting business value, dev effort, and technical readiness into a decision. Teams default to gut feel or HIPPO (highest-paid person's opinion). Specs pile up with no clear "build this next."

Proposed approach

A scoring template per spec (impact, effort, risk, dependency count) that Claude populates using graph analysis + spec content, producing a ranked backlog view.

🔍

Discovery gap

User interviews, usability tests, and research sessions produce notes — but those notes never make it into speckit. The bridge from "we learned this about users" to "here's the requirement" is manual and lossy. Specs get written without the discovery that should drive them.

Proposed approach

A discovery-to-spec bridge: paste interview notes or research summaries, Claude synthesizes themes and maps them to spec sections (problem statement, scenarios, success criteria).

💬

Meeting decisions evaporate

Design decisions get made in calls. Architecture choices get agreed in standups. Feature scope gets changed in reviews. None of it makes it into the spec or constitution. The only record is "someone remembers" — until they leave or forget.

Proposed approach

A meeting-notes-to-spec extractor: paste or upload call notes, Claude identifies decisions that should amend the spec or constitution, and proposes the changes for review.

✍️

Spec approval is informal

A PM writes a spec. A developer reads it. A stakeholder glances at it. Approval is a Slack "👍" or an assumed silence. There's no audit trail, no explicit sign-off gate, and no way to know who approved what version. When something goes wrong, "the spec was approved" is unverifiable.

Proposed approach

A spec status header (Draft → In Review → Approved) with a required approver list per spec, tracked in the spec file itself. Claude refuses to move to speckit-plan until approvers are listed.

👁️

Spec-blind code review

Code reviewers evaluate PRs against style guides and tests — not against the spec. A PR can technically pass all checks while silently violating requirements REQ-004 and REQ-007. Nobody knows until UAT or production.

Proposed approach

A GitHub Action that posts the relevant spec's requirements as a checklist on every PR that touches a specced feature, so reviewers can tick off what they verified.

🧑‍💼

UAT without a template

Stakeholder acceptance testing is ad-hoc. Testers click around. Issues get reported as "it feels wrong." There's no checklist derived from the spec's success criteria, no sign-off record, and no traceability from what was tested to what was required.

Proposed approach

A UAT checklist auto-generated from spec success criteria and scenarios. Each item references the requirement ID. Stakeholders check items off, and Claude summarizes the sign-off status.

🚀

Deployment disconnected from spec

Deployment checklists are generic: "run migrations, check logs, notify team." They have no awareness of what spec is shipping. A feature with a critical payment flow gets the same pre-deploy checklist as a UI label rename.

Proposed approach

A spec-aware deployment checklist generator: Claude reads the spec and generates a checklist specific to the feature's risk areas, dependencies, and constitution principles.

📈

Success criteria never measured

Specs define success criteria: "100% of plans have a supervisor before going Active." But after launch, nobody checks. The spec lives in the repo. The metrics live in Amplitude. Nobody connects them. Features ship and succeed or fail silently.

Proposed approach

A post-launch review template that pairs each spec success criterion with a metric query. Claude reads the criteria and suggests what to measure and where to find it in your analytics tool.

🕸️

No dependency map between specs

Spec A introduces a new status enum. Spec B adds a workflow that depends on it. Neither spec references the other. When spec A changes, spec B is never updated. The dependency was implicit — it should be explicit.

Proposed approach

A "Dependencies" section in each spec template that references other spec IDs. Claude uses the merged graph to suggest likely dependencies when writing a new spec.

⚠️

Technical debt is undocumented

Every team makes pragmatic shortcuts: "we'll do it properly next sprint." These shortcuts deviate from the spec but are intentional. Without a record, the shortcut becomes permanent — the next developer doesn't know it was a cut corner, not a design decision.

Proposed approach

A debt-spec format: when speckit-converge finds a divergence that's intentional, Claude helps document it as a debt record with context, risk level, and a planned resolution date.

🧑‍🎓

No onboarding path for new team members

A new developer joins. They have access to the repo, the graph, and the specs. But there's no guided path: "here's what this product does, here are the important domains, here are the non-negotiable principles, here's how we work." The institutional knowledge is present but not surfaced.

Proposed approach

A brainify onboarding mode: given a graph, constitution, vocabulary, and a set of specs, Claude generates a personalized onboarding guide for a new team member's specific role.

📝

Release notes are manual

CHANGELOG.md gets updated because the constitution requires it. Specs define what was built. But release notes for customers, stakeholders, or app stores are written from scratch every time — even though the spec and changelog contain everything needed.

Proposed approach

A release notes generator: Claude reads the CHANGELOG entries and linked specs for a release tag, and produces audience-appropriate release notes (technical, business, customer-facing) in one pass.

Every open problem is a contribution opportunity

You don't need to solve a whole problem. A good template, a working skill step, or a tested workflow that covers one scenario is enough to move it forward. Small PRs welcome.
